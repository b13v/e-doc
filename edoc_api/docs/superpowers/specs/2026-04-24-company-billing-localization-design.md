# `/company/billing` Localization Design

## Problem

The tenant-facing billing page at `/company/billing` is still rendered in English. That includes:

- visible page copy in the billing template
- labels and helper text around outstanding invoices and upgrade/payment flows
- success and error flash messages triggered by billing page actions

This is inconsistent with the rest of the tenant UI, which is expected to be localized to Russian and Kazakh.

## Goal

Make the tenant billing experience on `/company/billing` fully localized in Russian and Kazakh without changing billing logic or admin billing behavior.

## Success Criteria

- All static tenant-facing copy on `/company/billing` is localized through `gettext`.
- Flash messages triggered by `/company/billing` actions are localized through `gettext`.
- Russian locale shows Russian text.
- Kazakh locale shows Kazakh text.
- Billing calculations, plan codes, invoice statuses, and routing behavior remain unchanged.
- `/admin/billing/...` remains out of scope and unchanged.

## Scope

In scope:

- `lib/edoc_api_web/controllers/billing_html/show.html.heex`
- `lib/edoc_api_web/controllers/billing_html_controller.ex`
- any small helper functions needed to localize billing page labels cleanly
- `priv/gettext/ru/LC_MESSAGES/default.po`
- `priv/gettext/kk/LC_MESSAGES/default.po`
- controller/template tests covering localized tenant billing output and flashes

Out of scope:

- admin billing pages
- billing domain data model changes
- invoice/payment business rules
- changes to plan limits, overdue logic, or monetization policy

## Recommended Approach

Localize the billing page at the presentation layer only.

- Replace hardcoded English strings in the HEEx template with `gettext`.
- Replace hardcoded English flash messages in the controller with `gettext`.
- Keep billing snapshot/domain structures unchanged.
- If page helpers currently emit English labels directly, move those labels to localized helper functions in the web layer rather than the billing domain layer.

This keeps locale-aware logic where it belongs and avoids mixing translations into the billing context.

## UX Behavior

### Page Copy

The following categories of copy on `/company/billing` should be localized:

- blocked/overdue banners
- section headings
- plan/status labels shown to tenants
- upgrade prompt copy
- outstanding invoice empty state
- payment form labels and button text
- instructional helper text

### Flash Messages

Localized flashes should cover the actions reachable from the page:

- upgrade invoice request success/failure
- payment submission success/failure
- any existing validation or not-found style feedback emitted by the billing HTML controller

## Architecture

### Template Layer

`show.html.heex` becomes the source of localized page copy.

- All hardcoded English strings should be replaced with `gettext`.
- Interpolated strings should use named placeholders.
- Repeated labels such as status/plan display should call localized helper functions rather than embedding English literals.

### Controller Layer

`BillingHTMLController` should localize all `put_flash/3` messages.

- Success and error flashes should use `gettext`.
- Existing redirect targets remain unchanged.

### Helper Boundary

If plan/status labeling is currently produced by helper functions that return English strings, those helpers should be localized in the web layer. The billing context should continue returning raw plan codes and status atoms/strings.

## Testing Strategy

Add or update controller tests to prove the page is localized.

Required coverage:

- Russian locale renders localized `/company/billing` copy
- Kazakh locale renders localized `/company/billing` copy
- upgrade invoice action returns localized flash text
- payment submission action returns localized flash text

Tests should assert for stable localized substrings rather than full-page snapshots.

## Risks

- Some labels may be generated by helper functions rather than inline template copy, so missing one helper can leave partial English behind.
- Existing tests may assert English flash text and will need to be updated to locale-aware expectations.
- Shared helpers used elsewhere must be changed carefully so admin billing pages do not unintentionally localize if they are meant to remain English-only.

## Acceptance

The work is complete when:

1. `/company/billing` shows no tenant-facing English UI copy in Russian or Kazakh locales.
2. Billing-related flashes from that page are localized in Russian and Kazakh.
3. Relevant controller tests pass with localized expectations.
4. Admin billing pages remain unchanged.
